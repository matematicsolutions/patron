**Cel**: udowodnic ze sciezka audytora dziala w przegladarce end-to-end. Wszystkie 3 ADR-y dotykaja tego samego flow (UI viewer -> eksport pack -> force compute fallback) i sa zazebione - testujemy razem.

**Szacowany czas**: 15-20 min (po jednorazowej konfiguracji 5-10 min).

**Wymagania**:
- Supabase dziala (sprawdz w panelu Supabase czy projekt online)
- Backend i frontend zbudowane (`npm install` zrobione w obu)

### 0.3 (Opcjonalne) Wygeneruj testowe eventy w audit_log

Smoke test pokaze pusta tabele jezeli audit_log jest pusty. Najszybciej:

**Wariant A** - przez UI Patrona (jezeli juz uzywales chatu):

Po prostu odpal aplikacje (Krok 1), wykonaj kilka wiadomosci w chacie. Kazda wiadomosc generuje `chat.message.user` + `chat.message.assistant` w audit_log.

**Wariant B** - bezposrednio przez Supabase Studio (jezeli chcesz zacząć od zera):

Otworz Supabase Studio -> SQL Editor, wklej:

```sql
-- 5 syntetycznych eventow testowych (NIE PRODUKCJA)
insert into audit_log (event_type, actor_user_id, payload, prev_hash, hash, ts)
values
  ('chat.message.user', null, '{"q":"test pytanie 1"}'::jsonb, repeat('0', 64), encode(sha256('test1'::bytea), 'hex'), now() - interval '5 minutes'),
  ('chat.message.assistant', null, '{"a":"test odpowiedz 1"}'::jsonb, encode(sha256('test1'::bytea), 'hex'), encode(sha256('test2'::bytea), 'hex'), now() - interval '4 minutes'),
  ('input_security_scan', null, '{"verdict":"clean"}'::jsonb, encode(sha256('test2'::bytea), 'hex'), encode(sha256('test3'::bytea), 'hex'), now() - interval '3 minutes'),
  ('mcp_security.gateway', null, '{"decision":"allowed-clean","tool":"saos__search"}'::jsonb, encode(sha256('test3'::bytea), 'hex'), encode(sha256('test4'::bytea), 'hex'), now() - interval '2 minutes'),
  ('rodo.export', null, '{"user_id":"test"}'::jsonb, encode(sha256('test4'::bytea), 'hex'), encode(sha256('test5'::bytea), 'hex'), now() - interval '1 minute');
```

**UWAGA**: To ZLAMIE hash chain (sztuczne hashe nie sa policzone z payload). OK dla smoke testu UI - **NIE rob tego w produkcji**.

## Krok 4 - Test eksportu audit pack (ADR-0047)

W otwartym overlay AuditEventDetail (Scenariusz 3.3):

**Scenariusz 4.1 - eksport gdy istnieje Merkle root**:

Wstepnie musi byc Merkle root pokrywajacy event. Jezeli nie odpaliles wczesniej `npm run merkle:trigger` w backend - przejdz do Scenariusza 4.2 (force compute fallback to ZAMKNIE TO).

- Klik button "Pobierz audit pack (JSON)"
- Stan przelacza sie na "Przygotowuje pack..." z spinnerem
- Po <1 sek download blob ze nazwa `audit-pack-event-{id}-{YYYYMMDD}.json`

**Expected pliku**:
```json
{
  "schema_version": "1.0",
  "pack_kind": "audit_event_export",
  "exported_at": "2026-05-27T...",
  "exporter": { "user_id": "...", "email": "twoj.email@..." },
  "event": { ... },
  "merkle_proof_bundle": { ... },
  "verifier_instructions": { ... },
  "integrity": { "algorithm": "SHA-256", "canonical_sha256": "..." }
}
```

**Scenariusz 4.2 - eksport gdy BRAK Merkle root (ADR-0048 fallback)**:

Jezeli wszystkie eventy sa swieze i auto-trigger ADR-0036 jeszcze nie zadzialal:

- Klik "Pobierz audit pack" -> stan przelacza sie na **needs-compute** (amber alert box)
- Tekst: "Event nie jest jeszcze pokryty przez Merkle root. Wymus compute..."
- Pojawia sie **secondary button "Wymus compute root i ponow eksport"** z ikona ShieldCheck

- Klik secondary button -> stan przelacza sie na **computing** z spinnerem
- POST /api/audit/merkle/compute-now odpala sie w tle, response z `{computed: true, root: ...}`
- **Auto-retry** GET /api/audit/export/:eventId
- Download blob z audit pack JSON

**Scenariusz 4.3 - compute gdy brak nowych eventow (edge case)**:

Jezeli klikales secondary button dwa razy pod rzad (lub odpaliles `npm run merkle:trigger` wczesniej w tej sesji):

- Drugi klik daje stan **failed** z tekstem: "Brak nowych eventow do compute - audit_log pusty od ostatniego roota."
- To NIE jest blad - to legalny stan systemu

**Troubleshooting**:
- Button "Pobierz audit pack" nie pojawia sie - sprawdz w przegladarce DevTools czy `audit-event-detail.tsx` zaimportowal `AuditExportButton`.
- 403 z `/api/audit/export/...` - whitelist email nie pasuje, sprawdz Krok 0.1.
- 500 z `/api/audit/merkle/compute-now` - Supabase niedostepne, sprawdz `backend/.env` `SUPABASE_URL` + `SUPABASE_SECRET_KEY`.

## Krok 5 - Test offline verifier CLI (ADR-0047)

W terminalu (backend), z pobranym plikiem `audit-pack-event-123-20260527.json`:

```bash
cd C:/Users/Wieslaw/patron/backend
npm run audit:verify-pack -- ~/Downloads/audit-pack-event-123-20260527.json
```

(podmień path na faktyczna lokalizacje pobranego pliku)

**Expected output**:

```
Plik: /home/.../audit-pack-event-123-20260527.json
schema_version: 1.0
pack_kind: audit_event_export
exported_at: 2026-05-27T...
exporter: twoj.email@... / uuid
event_id: 123
event_type: chat.message.user
event_ts: 2026-05-27T...

[1/2] integrity SHA-256: OK (<hash>)
[2/2] Merkle proof: OK (event_id=123 odtwarza merkle_root <hash>)

Wynik: audit pack zdrowy (oba checki PASS)
```

Exit code: `0` = oba checki PASS.

**Scenariusz 5.1 - tampering detection**:

Edytuj pobrany plik JSON, zmień jedna cyfre w `event.payload_masked` (np. dodaj "X" w stringu). Zapisz. Uruchom ponownie:

```bash
npm run audit:verify-pack -- <plik>
```

**Expected**:
```
[1/2] integrity SHA-256: FAIL
      audit-pack: canonical_sha256 mismatch - pack zostal zmodyfikowany po eksporcie
      expected: <hash1>
      actual:   <hash2>
```

Exit code: `1`. To dokladnie to czego sie spodziewamy - integrity wykrywa modyfikacje.
